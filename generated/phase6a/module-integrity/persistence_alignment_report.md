## 2. ID Generation Strategy Analysis

### 2.1 UUID-as-String IDs (3 entities)

`UserJpaEntity.userId`, `DocumentLegalJpaEntity.documentId`, `IngestionJobJpaEntity.ingestionId` all use `String` UUID with no `@GeneratedValue`. The UUID must be generated by the application layer before calling `save()`.

**RUNTIME_RISK — No enforcement:** If a caller forgets to set the ID before persisting, JPA will attempt to INSERT with `userId = null`, which will fail with a database `NOT NULL` constraint violation. The error message will be confusing because JPA expects the ID to be set externally and doesn't validate it before the INSERT.

**Recommendation:** Add compact-constructor-equivalent validation in entity constructors:
```java
public UserJpaEntity(String userId, ...) {
    Objects.requireNonNull(userId, "userId must not be null");
    if (userId.length() != 36) throw new IllegalArgumentException("userId must be UUID format");
    ...
}
```

**Assessment:** The `protected UserJpaEntity() {}` no-arg constructor (JPA-required) is correctly `protected` — clients must use the parameterized constructor. This is PASS.

### 2.2 Oracle SEQUENCE for UserRoleJpaEntity

```java
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "user_role_seq")
@SequenceGenerator(name = "user_role_seq", sequenceName = "NOTARIST.USER_ROLE_MAP_SEQ", allocationSize = 1)
```

**Assessment:** `allocationSize = 1` is safe but slower than higher allocation sizes. For a role mapping table (low volume), `allocationSize = 1` is appropriate. No issue.

**Schema dependency:** The sequence `NOTARIST.USER_ROLE_MAP_SEQ` must exist in Oracle DDL. Liquibase migration must CREATE SEQUENCE before the entity is first persisted.

## 3. JPQL Query Safety Analysis

### 3.1 DocumentLegalJpaRepository — Raw String Filter Parameters

```java
@Query("""
    SELECT d FROM DocumentLegalJpaEntity d
    WHERE d.tenantId = :tenantId
      AND (:documentType IS NULL OR d.documentType = :documentType)
      AND (:status IS NULL OR d.status = :status)
      AND (:maxClearanceLevel IS NULL OR d.classificationLevel IN :allowedLevels)
    """)
List<DocumentLegalJpaEntity> findByTenantIdWithFilters(
    @Param("documentType") String documentType,
    @Param("status") String status,
    @Param("maxClearanceLevel") Integer maxClearanceLevel,
    @Param("allowedLevels") List<String> allowedLevels,
    ...);
```

**INTEGRITY_RISK — Raw String enum values in JPQL:**  
`documentType`, `status`, `classificationLevel` are stored as `String` in the entity (per Section 2 of `enum_persistence_audit.md`). The filter parameters `documentType` and `status` are also `String`. This means:
1. The caller passes `"INDEXED"` as a string — no type check that it's a valid `DocumentStatus` value
2. A typo like `"INDEXD"` silently returns no results instead of throwing an error
3. When enum constants are renamed, this query silently breaks

**Remediation:** Once enum fields are converted to typed fields (see `enum_persistence_audit.md`), update queries to compare `DocumentStatus` enum values. The `allowedLevels` list should be `List<ClassificationLevel>` or `List<String>` of `.name()` values.

### 3.2 IngestionJobJpaRepository — Hardcoded Status Literal

```java
@Query("... WHERE j.pipelineStatus = 'FAILED' ...")
```

**INTEGRITY_RISK:** Hardcoded string literal `'FAILED'` in JPQL. If `PipelineStatus.FAILED` is renamed or the FAILED status is restructured, this query silently breaks. It will return zero results — no compile error, no test failure until production.

**Remediation:** Replace with:
```java
@Query("... WHERE j.pipelineStatus = :#{T(com.notarist.ingest.domain.model.PipelineStatus).FAILED.name()} ...")
// Or use named constant
private static final String FAILED_STATUS = PipelineStatus.FAILED.name();
```

Or better: once field becomes enum-typed, use:
```java
@Query("... WHERE j.pipelineStatus = com.notarist.ingest.domain.model.PipelineStatus.FAILED ...")
```

## 4. @Lob on Oracle CLOB

### IngestionJobJpaEntity — stageHistoryJson

```java
@Lob
@Column(name = "STAGE_HISTORY", nullable = false)
private String stageHistoryJson;
```

**INTEGRITY_RISK — Oracle CLOB + Hibernate behavior:**  
On Oracle, `@Lob` on a `String` field maps to `CLOB`. This is intentional for large JSON. However:

1. Oracle `CLOB` values cannot be compared with `=` in WHERE clauses — only `LIKE` or `SUBSTR()`. This limits JPQL filtering on this column (not currently filtered — OK).
2. Hibernate lazy-loads LOBs by default in some configurations. With Oracle OJDBC11, ensure `oracle.jdbc.defaultLobPrefetchSize` is set appropriately to avoid N+1 LOB fetch issues.
3. The Liquibase DDL must declare this column as `CLOB`, not `VARCHAR2`.

**Severity:** RUNTIME_RISK — not a compile blocker, but incorrect LOB prefetch configuration causes significant performance degradation under load.

## 5. Missing JPA Annotations

### 5.1 No @Version on any entity

None of the 4 entities has `@Version` for optimistic locking.

**LATENT_RISK:** In a concurrent ingest system where multiple workers may update `IngestionJobJpaEntity.pipelineStatus` simultaneously (retry scheduler + active worker), a missing `@Version` means no optimistic lock check — lost updates are possible.

**Recommended:** Add `@Version private Long version` to `IngestionJobJpaEntity` at minimum.

### 5.2 Missing updatable=false on immutable columns

`createdAt` columns in all entities are declared with `nullable = false, updatable = false` — **PASS**.  
`checksumSha256` in `DocumentLegalJpaEntity` and `IngestionJobJpaEntity` should also be `updatable = false` (checksums are immutable after creation). Currently missing this annotation.

### 5.3 No @Column(columnDefinition) for Oracle-specific types

For Oracle 19C, some column definitions benefit from explicit type declarations:
- UUID fields: `VARCHAR2(36)` — not declared, defaults to JDBC mapping
- TIMESTAMP fields (Instant): Oracle needs `TIMESTAMP(6) WITH TIME ZONE` — not declared

These are LATENT_RISK items — Hibernate will use default DDL generation which may or may not match the Liquibase-created schema.

## 6. Oracle vs PostgreSQL Routing

### Persistence Layer Split

| Data | Database | Framework | Module |
|---|---|---|---|
| Users, Roles | Oracle 19C — NOTARIST schema | JPA / Hibernate | notarist-auth |
| Documents | Oracle 19C — NOTARIST schema | JPA / Hibernate | notarist-document |
| Ingestion jobs | Oracle 19C — NOTARIST schema | JPA / Hibernate | notarist-ingest |
| Session tokens | PostgreSQL | Spring JDBC (`JdbcTemplate`) | notarist-auth |
| Ingestion queue | PostgreSQL — SKIP LOCKED pattern | Spring JDBC | notarist-ingest |
| Dead letter queue | PostgreSQL | Spring JDBC | notarist-ingest |
| BM25 search index | PostgreSQL — pg_trgm / tsvector | Spring JDBC | notarist-search |
| Vector metadata | Qdrant | Qdrant Java client | notarist-infra |

**Assessment: PASS — separation is clean. Oracle for transactional entities; PostgreSQL for queue/search/session.**

### DataSource Configuration Risk

`notarist-web` needs two `DataSource` beans:
- `oracleDataSource` — for JPA EntityManager
- `postgresDataSource` — for JDBC operations

**RUNTIME_RISK:** Without explicit `@Primary` on the Oracle DataSource, Spring JPA auto-configuration may pick the wrong one for the `EntityManagerFactory`. This must be explicitly configured in `notarist-web` or `notarist-infra`.

Verify `NotaristApplication` or a `DataSourceConfig` class properly configures:
```java
@Bean @Primary @ConfigurationProperties("spring.datasource.oracle")
public DataSource oracleDataSource() { ... }

@Bean @Qualifier("postgres") @ConfigurationProperties("spring.datasource.postgres")
public DataSource postgresDataSource() { ... }
```

## 7. Migration Tool Ownership

### Dual Migration Tools — Confirmed Risk

| Tool | Manages | Status |
|---|---|---|
| Liquibase | Oracle DDL (`NOTARIST.*` tables, sequences) | Should be authoritative for Oracle |
| Flyway | PostgreSQL DDL (queue, session, BM25 tables) | Should be authoritative for PostgreSQL |

**RISK-L2 from Phase 6A.1 — NOT RESOLVED.**  
This is acceptable if each tool has exclusive ownership of its target database. Risk becomes BLOCKER if either tool attempts to manage the other's database.

**Verify in `notarist-web` configuration:**
```yaml
spring:
  liquibase:
    url: ${ORACLE_URL}
    enabled: true
  flyway:
    url: ${POSTGRES_URL}
    enabled: true
```

If `spring.flyway.url` is not set, Flyway defaults to `spring.datasource.url` — which may be Oracle. This would cause Flyway to attempt Oracle DDL with PostgreSQL-syntax migrations.
